# resizable-columns Implementation Plan

**Goal:** RETARGET the already-built feature to match the corrected spec. The 3-way size preset moves OFF the folder sidebar and ONTO the **account rail** (`SidebarSize` → `RailSize` {small/medium/large}, gaining `tileSize` and renaming `showsLabels` → `showsNames`; small = today's 56/38 icon-only DEFAULT, medium = bigger icon-only tiles, large = bigger tiles + account names). The folder sidebar gets TWO simpler controls instead: a two-state `sidebarLabelsVisible` toggle (reusing the already-built compact render) and a draggable `sidebarWidth` via a NEW sidebar↔list handle. The list↔reader handle (`listWidth`, `clampListWidth`, built `ListDragHandle`) is UNCHANGED. All four persisted settings additive: a user who touches nothing sees today's exact layout (rail 56/38 icon-only, sidebar labels-visible @232, list @380).

**Architecture:** All sizing logic stays in the pure, view-free seams in `MMail/State/LayoutSizing.swift` (`RailSize` enum + `clampSidebarWidth` + `clampListWidth`), unit-tested in isolation (swift-testing). `AppModel` holds FOUR `@Published` values (`railSize`, `sidebarLabelsVisible`, `sidebarWidth`, `listWidth`) and persists them via the existing `persistTweaks`/`UserDefaults` pattern, with a targeted single-key write for each per-drag path. The views read those seams; both the "Account Rail Size" submenu and the "Show Folder Labels" toggle are stateful SwiftUI added directly in `MMailApp`'s `CommandGroup(after: .sidebar)` (NOT routed through the pure `MenuModel`/`buildCommands()`). The new sidebar↔list drag handle is a private inline `struct` in `RootView.swift` mirroring the built `ListDragHandle`.

**Test Methodology:** e2e-first (from `.harness.yaml`).

**Swift / worktree caveats (read before building):**
- Working tree: worktree `.worktrees/resizable-columns`, branch `feat/resizable-columns`, isolated `.harness.yaml` (`current_feature: resizable-columns`). dark-engine has since merged to its main, so the main checkout is now FREE for the deferred test run (T-D1).
- **Per-phase gate = `xcodebuild -project MMail.xcodeproj -scheme MMail -configuration Debug build-for-testing CODE_SIGNING_ALLOWED=NO`** run FROM THE WORKTREE. This compiles the app AND the `MMailTests` bundle without launching, so it does NOT hit the worktree `xcodebuild test` LaunchServices hang (see `project_worktree-xctest-hang`). It is the gate for every code phase.
- **The actual test RUN (`xcodebuild … test`) is DEFERRED to /verify on the main checkout** (T-D1) — `test` hangs from a worktree, and a detached-HEAD checkout of the feature sha on the main checkout sidesteps both the hang and the "branch checked out elsewhere" lock. Tests are AUTHORED now (Phase A) and run at /verify; they are expected to PASS.
- **NO `xcodegen generate` needed this time** (drops the original plan's T003 regen task). `LayoutSizing.swift`, `LayoutSizingTests.swift`, and `RootView.swift` all already exist; the new sidebar↔list handle is an inline `private struct` in the existing `RootView.swift`. No `.swift` file is added or removed, so `project.pbxproj` stays clean — this is a simplification vs. the original plan, which added two new files and needed a regen.
- **Atomicity hazard:** the `SidebarSize` → `RailSize` rename and the `LayoutDefaultsKey` key swap touch `LayoutSizing.swift`, its tests, `AppModel.swift`, `SidebarView.swift`, `MMailApp.swift`, AND `RootView.swift` (the `value: model.sidebarSize` animation). The tree will NOT compile until every `sidebarSize`/`SidebarSize`/`showsLabels`/`setSidebarSize`/`cycleSidebarSize` reference is migrated. Phase A renames the seam + tests; the first GREEN gate is therefore at the END of Phase C (T-C7), not per-symbol. Phases A/B compile-gate the seam+model in isolation only because the views still reference the OLD names until Phase C — so **build Phases A→C as one continuous unit; do not expect a green app build until T-C7.** (The seam+tests bundle from Phase A compiles against the renamed seam; the APP target only goes green once the views are migrated in Phase C.) After migration, NO symbol named `sidebarSize`, `setSidebarSize`, `cycleSidebarSize`, `SidebarSize`, `mmail.sidebarSize`, `showsLabels`, or `loadSidebarSize` survives anywhere.
- Build dispatched to a single Opus subagent (per CLAUDE.md). Reviews are opposite-model (Sonnet).
- **Subagent branch-switch guard:** the build subagent's Bash can switch the shared working-tree branch (see `subagent_branch-switch`). Before EVERY commit it MUST verify `git branch --show-current` == `feat/resizable-columns` AND CWD == `.worktrees/resizable-columns`, re-asserting `git checkout feat/resizable-columns` if needed.

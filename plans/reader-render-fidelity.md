# reader-render-fidelity Implementation Plan

**Goal:** The reader renders an email body the way its sender composed it — on a stable opaque-white surface regardless of app theme, with inline `cid:` images shown inline (not as broken boxes plus stray attachments).

**Architecture:** Two independent fixes sharing one feature.

- **Bug A (white surface)** is two view-layer edits with one pure seam: (A1) the `HTMLMessageView.wrapped(_:)` head builder forces `color-scheme: only light` + an opaque white body background; (A2) the reader's plain-text-only `else` branch — which never passes through `HTMLMessageView` — gets its own white-surface treatment (fixed dark text on a white background, theme-independent). A1's head construction is extracted to a pure function so it is unit-testable without a WebView.

- **Bug B (inline CID images)** is a parse→render pipeline. (B1) the MIME parser captures each part's `Content-ID` onto the in-memory `MIME.Attachment` (a new optional `contentID: String?` — non-Codable, so no cache-schema change). Two pure seams: (B2) "is CID token referenced by `<img src="cid:…">` in the HTML?" predicate; (B3) the `cid:`→`data:<mime>;base64,…` rewrite over (HTML, CID→part map). (B4) carry the CID part BYTES (not just `AttachmentMeta` metadata — the bytes are discarded after parse today and must be plumbed through, including out of the prefetch loop where the `MIME.Attachment`s otherwise go out of scope) from parse to render via a render-time-only, non-cache-serialized map keyed by `Email.id`, applying B3 just before `wrapped(_:)`. (B5) a SECOND pass at the `MIME.Attachment → AttachmentMeta` promotion boundary in `AppModel` drops parts whose CID is referenced inline (keeps unreferenced CID parts as attachments). (B6) a build-time check of the `blockRules` `.*` rule vs `data:` URIs, with a `data:`-scheme `ignore-previous-rules` carve-out as the contingent fix.

The pure seams (A1 head builder, B2 predicate, B3 rewrite) are early so each is independently unit-testable before any wiring. The `MIME.Attachment.contentID` field (B1) is in-memory only — the cache-serialized `Email`/`AttachmentMeta` types are UNCHANGED (SC-009).

**Visibility constraint (verified):** `HTMLMessageView.wrapped(_:)` is `private static` (`HTMLMessageView.swift:75`) and MUST STAY `private static` — it is only ever called from inside that view. The A1 head builder moves OUT to `CIDInlining.swift`, where it (and the B2 predicate + B3 rewrite) MUST be at least `internal` (the default — do NOT mark them `private`/`fileprivate`) so `@testable import MMail` in `MMailTests/CIDInliningTests.swift` can reach them. `HTMLMessageView.wrapped` then becomes a thin `private` call into the `internal` builder (T009).

**New-file recommendation:** add ONE new production file `MMail/Mail/CIDInlining.swift` housing the three pure seams (A1 head builder, B2 predicate, B3 rewrite + the CID→part map type), mirroring how `BodyCompleteness.swift` houses the `BodyFetch` seam and `ReaderImageLoadState.swift` houses the reader-HTML image seam. Rationale: keeps `HTMLMessageView.swift`/`MIME.swift` from absorbing test-only surface, and `@testable import MMail` reaches it cleanly. Plus ONE new test file `MMailTests/CIDInliningTests.swift`. Both new files require `xcodegen generate` ONCE (T002), then commit the regenerated `MMail.xcodeproj/project.pbxproj` — `project.yml` globs `sources: - path: MMail` and `- path: MMailTests` recursively (verified `project.yml:35,66`), so new files under either tree are auto-included. The A1 builder is *referenced from* `HTMLMessageView.wrapped`, A2 stays inline in `ReaderView`, and the B4 render-time rewrite is called from `ReaderView` before constructing `HTMLMessageView`.

**Test Methodology:** e2e-first (`.harness.yaml`). For this macOS SwiftUI app: failing XCTests first over the pure seams (A1 head, B2 predicate, B3 rewrite, plus a pre-feature-cache decode test for SC-009), implementation drives them green; then `manual-exploration` is the true end-to-end gate (WebView render + real ETHS-style signature email + remote-image blocking — none exercisable headlessly). Always-on per `.harness.yaml`: type-driven, manual-exploration, review.

  **`openAttachment` is NOT in scope (finding 4):** the third `MIME.parse` site, `openAttachment` (`AppModel.swift:2867`), looks up a part by `meta.filename` over the already-filtered `[AttachmentMeta]`; after B5 a referenced CID part is absent from that list, so this path can never be asked to download it — correct and safe.   - Run: BUILD — Expected: PASS.

- [ ] **T007 (SC: 003, 006, 009): Carry CID→BYTES to render without serializing it** — The referenced CID part BYTES (`MIME.Attachment.data`) are discarded after parse today, and they are what `data:` rewriting (B3) needs — `AttachmentMeta` keeps only `size`, so metadata is NOT enough. Plumb the referenced CID→part map (from T006) — a `[String: InlinePart]` where `InlinePart` carries `(mimeType: String, data: Data)` (the actual bytes, base64-encoded at render time) — from parse to the reader via a render-time-only store NOT on the cache-serialized `Email`. Recommended shape (type-driven build finalizes it): a dictionary on `AppModel` keyed by `Email.id` → `[String: InlinePart]`, populated at the parse sites alongside the body assignment, read by `ReaderView`. In-memory only — NEVER written to `MailCache`, so the cache keeps raw `cid:` HTML (SC-006) and no `Email`/`AttachmentMeta` key is added (SC-009).

  **The bytes MUST survive to the MainActor apply (finding 2 — feasibility gap).** The prefetch `parsedByUID` tuple (`AppModel.swift:2681`) today holds only scalars/strings/`[AttachmentMeta]`; the parsed `MIME.Attachment` objects (which hold `.data`) go OUT OF SCOPE at the end of the `for (uid, data)` loop, so the bytes are GONE before the MainActor apply loop. The earlier "mirror how `complete: Bool` was added" analogy is WRONG: `complete` is a scalar; this is a potentially large bytes map. Therefore the `parsedByUID` tuple MUST be extended to carry the referenced CID→`InlinePart` map (the actual bytes), built inside the loop while the `MIME.Attachment`s are still in scope, then assigned per-id in the MainActor loop (`~2693-2711`). For the open site, set the map in the success `MainActor.run` block (`~2788-2807`).

  **Prefetched-and-complete hazard (finding 2).** If a prefetched message already has `bodyComplete == true`, `loadBodyIfNeeded` returns early (`BodyFetch.needsFullFetch` is false — `AppModel.swift:2755`), so the open path NEVER re-fetches and never re-populates the map. The prefetch path MUST therefore populate the id-keyed CID-bytes map itself (not rely on a later open-fetch), so a prefetched-complete message still has a populated render-time map. (The 64 KB prefetch cap means a CID-bearing message is often truncated/incomplete and re-fetched on open — but do NOT rely on that; populate from prefetch unconditionally so the complete-on-prefetch case is covered.)

  **serverSearchResults render path (finding 3 — verified, known-safe).** `ReaderView` renders `model.selectedEmail` (`ReaderView.swift:10`); `selectedEmail` resolves through `filteredEmails`/`visibleEmails`, which in active server-search returns the `serverSearchResults` COPIES (`AppModel.swift:334`). So a message opened from server search renders via the `serverSearchResults[j]` copy, NOT `emails[i]`. This is SAFE because the CID-bytes map is keyed by `Email.id`, and the `serverSearchResults[j]` copy shares the SAME `id` as the `emails[i]` row. The open-path map population (keyed by `id`) therefore covers BOTH copies with a single write — no separate search-keyed map is needed. (Confirmed by code: `selectedEmail` = `AppModel.swift:377-378`, `visibleEmails` search branch = `334`, `ReaderView` body source = `ReaderView.swift:10`.)

  **Eviction (finding 5).** The map currently has NO eviction hook (none exists today): it grows by ONE entry per opened/prefetched CID-bearing message and is NOT evicted for the session — an ACCEPTED, bounded-for-now memory leak (each entry is the inline image bytes for one message). State this plainly; do not add an eviction mechanism in this feature. Bounded by the count of CID-bearing messages touched in a session.
  - **Files:** `MMail/State/AppModel.swift` (map property; prefetch `parsedByUID` tuple build + MainActor assignment; open-path assignment)
  - Run: BUILD — Expected: PASS.

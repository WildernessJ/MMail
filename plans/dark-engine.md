# dark-engine Implementation Plan

**Goal:** When the app is in dark mode, the reader renders an email body in a native-dark theme (dark backgrounds, light text, images and brand colors preserved) by running the vendored DarkReader engine in-page over the existing white substrate — with a per-message "Show original" escape hatch and a matching dark treatment for the plain-text branch.

**Architecture:** One feature layered onto the shipped `reader-render-fidelity` white surface. The substrate (`ReaderHTML.wrappedDocument`, `MMail/Mail/CIDInlining.swift:32`, emitting `:root { color-scheme: only light; }` at `:37`) is UNCHANGED; this feature transforms that document IN-PAGE after load. Five layers:

- **Spike (feasibility, load-bearing).** Vendor a real UMD `darkreader.js` (MIT) as a bundled resource under `MMail/Resources/`, register it via `xcodegen generate`, load it from `Bundle.main` at runtime, inject it into the loaded `<body>` and call `DarkReader.enable(theme)` from `HTMLMessageView` when `model.dark`. The spike's gate is a LIVE visual confirmation that a real email darkens (images intact, signature logo NOT inverted) WITH `color-scheme: only light` left intact. If infeasible, the build HALTS and escalates — no silent CSS-filter fallback.

- **Pure seams (new internal surface).** Two pure functions in `ReaderHTML` (existing enum in `CIDInlining.swift`, already `internal`): (1) `shouldApplyDark(dark:showOriginal:)` — the `model.dark && !showOriginal` predicate; (2) `darkEnableScript()` — the fixed-dark-palette injection/`enable()` JS string builder. Both unit-testable without a WebView, in a new `MMailTests/DarkEngineTests.swift`.

- **HTML dark wiring + reactivity + height sequencing.** Thread the dark-apply decision into `HTMLMessageView` as a new `applyDark: Bool` input; react to its change in `updateNSView` change-detection. Implement the inject→`enable()`→re-measure `scrollHeight` sequence (modifying the load-completion path) so height reflects the transformed DOM. Toggle mechanism is IN-PLACE `DarkReader.enable/disable` via `evaluateJavaScript` (no reload flash), re-measuring height after — chosen over a full reload to avoid a flash on every dark↔light / Show-original toggle.

- **"Show original" per-message control.** New `@State private var showOriginal = false` on `ReaderContent` (`MMail/Views/ReaderView.swift:36`), reset per-message via the existing `.id(email.id)` (`:12`), mirroring `loadImages` (`:48`). A reader control toggles it; it feeds the dark-apply decision.

- **Plain-text SwiftUI dark treatment.** When `model.dark && !showOriginal`, the plain-text branch (`ReaderView.swift:335-351`, native `Text` — NOT a WebView) renders on a dark surface (~`#1A1A1A`) + light text; light / Show-original keeps today's dark-on-white.

**Test Methodology:** e2e-first (`.harness.yaml`). For this macOS SwiftUI app: failing XCTests first over the pure seams (the dark-apply predicate + the injection-script builder) plus a pre-feature-cache decode test for SC-010; implementation drives them green; then `manual-exploration` is the true end-to-end gate (WebView render, real emails, the DarkReader transform, height/no-clip, remote-blocking — none exercisable headlessly). Always-on per `.harness.yaml`: type-driven, manual-exploration, review.

**Commands (Swift, not the skill's Python templates):**
- BUILD = `xcodebuild -project MMail.xcodeproj -scheme MMail -configuration Debug build CODE_SIGNING_ALLOWED=NO`
- TEST  = `xcodebuild test -project MMail.xcodeproj -scheme MMail -destination 'platform=macOS' CODE_SIGNING_ALLOWED=NO`
- New files (the vendored `darkreader.js` resource + a new `MMailTests/DarkEngineTests.swift`) → run `xcodegen generate` ONCE then commit the regenerated `MMail.xcodeproj/project.pbxproj`. `project.yml` globs `sources: - path: MMail` and `- path: MMailTests` recursively, so a file under either tree auto-includes; a non-Swift extension under `MMail/` (`.js`) is auto-classified by XcodeGen as a bundle RESOURCE.
- **Anchors below are from the pre-change tree — RE-GREP before editing; they drift.** (SourceKit live diagnostics in this repo are known-false: "Cannot find type X in scope" for same-module types, "No such module 'Testing'" — trust the real `xcodebuild`, not the editor squiggles.)

**Visibility constraint (verified):** the pure seams live on the existing `enum ReaderHTML` in `MMail/Mail/CIDInlining.swift` (already documented `internal`, the default — see the file header `:11-12`). Do NOT mark the new functions `private`/`fileprivate` or `@testable import MMail` in `DarkEngineTests.swift` cannot reach them. `HTMLMessageView.wrapped(_:)` stays `private static` (`HTMLMessageView.swift:80`) — it is only called inside that view.

**New-file recommendation:** put the vendored engine at `MMail/Resources/darkreader.js` (a real UMD `darkreader.js` from a DarkReader release that exposes a global `DarkReader` with `enable`/`disable`, MIT — record version + provenance + the MIT notice in a top-of-file comment AND in the spike task). Only two new files (the `.js` resource + the test file) require `xcodegen generate`.

**Re-render mechanism (decided — spec left it to the plan):** IN-PLACE JS toggle. On a dark→light / Show-original change the view calls `DarkReader.disable()` (and re-`enable()` to re-darken) via `evaluateJavaScript` on the ALREADY-LOADED page, then re-measures `scrollHeight`. Rationale: a full reload (extending `lastHTML`/`lastBlock`/`lastProxyConfig` change-detection to re-run `load`) is correct but flashes the WebView white-then-dark on every toggle; the in-place toggle has no flash. The cost — height MUST be re-measured after the toggle settles — is handled by routing every transform through a single inject→`enable()`/`disable()`→measure helper (T008). A NEW input (`applyDark`) is still threaded into `updateNSView` so SwiftUI re-runs the view when the decision changes; that input drives the in-place toggle, NOT a reload.

**Out of scope (verified, intentionally not touched):** the remote-block / image-proxy / `^data:` carve-out content rules (`HTMLMessageView.blockRules` `:105-108`, `proxyAllowRules(for:)` `:134-150` — note the `^data:` carve-out and `color-scheme: only light` ALREADY shipped with `reader-render-fidelity`; this feature does NOT touch them), the CID-inlining path (`ReaderHTML.inlineCIDImages` / `model.inlineParts(for:)`, `ReaderView.swift:330`), which messages are fetched, body-completeness (`BodyFetch`), and the cache schema. NO cache-serialized type (`Email`, `AttachmentMeta`) gains a required key — the transform is render-time only. No System/Light/Dark auto-follow (keys off `model.dark` only). No theme customization / multiple dark themes (one fixed palette, bg ≈ `#1A1A1A`). No "disable inline email JS" hardening. No auto CSS-filter fallback.

**Anchor sweep (verified against the live tree, RE-GREP before editing):**

Consumed by `HTMLMessageView.wrapped` (`HTMLMessageView.swift:80`). `ReaderHTML.bodyTextColorHex`/`bodyTextColor` (the `#1A1A1A` single source of truth) at `CIDInlining.swift:18,22`.

## Phase C — HTML dark wiring + reactivity + height sequencing

> T008 and T009 are intermediate BUILD-green checkpoints with NO commit; they commit atomically at T010. - [ ] **T008 (SC: 002, 007): Thread `applyDark` into `HTMLMessageView` + a single inject→enable/disable→measure helper** — Add a new `var applyDark: Bool` input to `HTMLMessageView` (`HTMLMessageView.swift:47-55`). **Remove the throwaway `model.dark` input added in T003 — `applyDark` REPLACES it; the final view has `applyDark`, NOT both.** Add a `Coordinator` stored `lastApplyDark` mirroring `lastBlock` (`:86-88`). Replace the unconditional immediate height measure in `webView(_:didFinish:)` (`:196-202`) with a single private helper `applyDarkAndMeasure(_ web:)` that: (1) if `applyDark` — `evaluateJavaScript` the vendored engine define (T002) THEN `DarkReader.enable(...)` via `ReaderHTML.darkEnableScript()` (T007); then, in that `enable()` `evaluateJavaScript` COMPLETION, **schedule the `scrollHeight` read on the NEXT runloop turn — `DispatchQueue.main.async { … }`** (because DarkReader's `enable()` injects CSS + attaches a MutationObserver and its effect is NOT fully applied synchronously when the `evaluateJavaScript` completion fires; reading `scrollHeight` in the bare completion samples a PRE/MID-transform height and violates SC-007) — inside that scheduled block `evaluateJavaScript("document.body.scrollHeight")` and set `height` so the synchronously-injected styles and first layout have applied. **Build-time tunable:** if the LIVE T004/T015 verify shows height still samples early, escalate the settle from `DispatchQueue.main.async` → `asyncAfter(deadline: .now() + ~0.016–0.032)` (one frame); if still early, escalate to a double-`requestAnimationFrame` in the injected JS that posts the settled height back via a `WKScriptMessageHandler`. This measure SUPERSEDES the immediate measure, so a darkening layout shift never leaves a stale height. (2) if NOT `applyDark` — measure `scrollHeight` immediately (today's path, unchanged). `didFinish` calls this helper. Keep the proxy / non-proxy `load` paths and `loadHTMLString(body, baseURL: nil)` unchanged.

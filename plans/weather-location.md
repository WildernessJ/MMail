# weather-location Implementation Plan

**Goal:** Make the Home weather widget resolve a typed `"City, ST"` / `"City State"` / `"City, Country"` string to the right place (e.g. `"kingsville md"` → Kingsville, Maryland, never Texas), surface a "Couldn't find that city" alert on an explicit-submit no-match while preserving the prior weather, and guarantee structurally that a typed city NEVER falls back to IP geolocation — IP stays exclusive to the empty-city "Use my location" path.

**Architecture:** A new pure, network-free seam `WeatherQuery` (an `enum` namespace of static funcs over a plain `Candidate` value) does ALL parsing, the US-state-abbreviation→full-name table, region matching, and best-candidate selection (INV-1) — the only unit-tested part. `WeatherService.fetch(city:)` is refactored to return a three-case `WeatherResult` enum (`.success(WeatherInfo)` / `.notFound` / `.error`, INV-4): the geocoder now requests `count` ≥ 10, decodes the candidate list into `[WeatherQuery.Candidate]`, and delegates the choice to `WeatherQuery.bestMatch`; the typed-city path provably never reaches `geolocate()` (INV-3). `AppModel.refreshWeather` consumes the outcome and only raises a new `@Published var weatherNotFound` flag when the lookup was an EXPLICIT user submit AND came back `.notFound` (INV-5) — automatic launch/refresh never alerts. `HomeView` binds a new `@State` alert to that flag. `WeatherInfo` and the `UserDefaults` key are byte-unchanged (Non-Goals).

**Test Methodology:** e2e-first

## Phase A — Pure `WeatherQuery` seam (TDD: parse + state table + bestMatch)

> Phase goal: land the network-free, deterministic `WeatherQuery` seam and its Swift-Testing suite covering SC-1..SC-4 — `parse` (incl. the falsifiable multi-word trailing-token rule), the US-state-abbreviation table + case-insensitive region matcher, and the four-arm `bestMatch`. ### T1 — Failing tests first: `WeatherQuery` parse + match + bestMatch (RED) (SC-1, SC-2, SC-3, SC-4)
Author `MMailTests/WeatherQueryTests.swift` (new) in the Swift-Testing house style (`import Testing`, `@Suite`, `@Test`, `#expect`) BEFORE the implementation exists, so the file compiles only after T2. Cover, traced to the success criteria:
- **`parse` (SC-1):** `"Kingsville, MD"`→`(city:"Kingsville", region:"MD")` (whitespace collapsed); `"kingsville md"`→`("kingsville","md")` (trailing token is a recognized US-state abbrev); `"Paris"`→`("Paris", nil)`; `"Paris, France"`→`("Paris","France")`; **multi-word rule (the falsifiable one):** `"San Francisco"`→`("San Francisco", nil)` — assert region is `nil` AND city is NOT `"San"` (the trailing token `"Francisco"` is NOT a recognized region token); and `"San Francisco, CA"`→`("San Francisco","CA")` (comma form always safe). Add a no-region country-code space form is NOT required, but DO assert a non-region trailing word stays in the city.
- **state table + region matcher (SC-2):** abbrev expansion `"md"`→`"Maryland"`, `"tx"`→`"Texas"`, `"ny"`→`"New York"` case-insensitively; and the matcher returns true when a region (case-insensitively) equals a candidate's full `admin1`, its `country`, or its `country_code` (e.g. `"Maryland"`, `"United States"`, `"US"`, `"France"`/`"FR"`).
- **`bestMatch` four arms (SC-3, SC-4):** (1) region matches a candidate's `admin1` via abbrev → that candidate chosen (Kingsville/Maryland over Kingsville/Texas); (2) region given but NO candidate matches (`region:"zz"`) → `nil` (INV-2, never the top candidate); (3) no region (`nil`) + non-empty list → the FIRST/top candidate; (4) EMPTY candidate list (with or without region) → `nil`.
- **SC-4 framing:** assert the pure success-vs-not-found split (matched-`Candidate` vs `nil`) only — the error outcome is NOT in the seam.
- `Run (worktree authoring gate, after T2 makes it compile — see note):` `xcodebuild -project MMail.xcodeproj -scheme MMail -configuration Debug build-for-testing CODE_SIGNING_ALLOWED=NO`

### T2 — Implement `WeatherQuery` to green: types + parse + state table + matcher + bestMatch (GREEN) (INV-1, INV-2, SC-1..SC-4)
New file. Define the pure seam as an `enum WeatherQuery` namespace (no instances, no I/O):
- **`struct Candidate`** — the value the network layer decodes into and the seam consumes: `let name: String; let admin1: String?; let country: String?; let countryCode: String?; let latitude: Double; let longitude: Double`. (Plain value type; the geocode JSON → `Candidate` mapping lives in `WeatherService`, T4.)
- **`struct Parsed`** (or a `(city: String, region: String?)` tuple return) — chosen type: a named `struct Parsed { let city: String; let region: String? }` for clarity.
- **`static func parse(_ raw: String) -> Parsed`** — trim + collapse internal whitespace; COMMA form: everything before the LAST comma is the city, the segment after is the region (both trimmed); NO-COMMA form: split on whitespace, and treat the trailing token as the region ONLY when `isRegionToken(_:)` is true (known US-state abbrev OR known country code OR a full admin1/country name present in the recognized set) — otherwise the WHOLE string is the city (so `"San Francisco"`→city `"San Francisco"`, region `nil`). Empty/whitespace input → `Parsed(city:"", region:nil)`.
- **US-state table** — `static let stateAbbreviations: [String: String]` (lowercased key → full name) for all 50 states + DC; `static func expandRegion(_ region: String) -> String` returns the full state name for a known abbrev (case-insensitive) else the input unchanged.
- **`static func regionMatches(_ region: String, candidate: Candidate) -> Bool`** — case-insensitive: let `expanded = expandRegion(region)`; true if `expanded` equals `candidate.admin1`, OR `region`/`expanded` equals `candidate.country`, OR `region` equals `candidate.countryCode` (all `.caseInsensitiveCompare`/lowercased). This is where `"md"`→Maryland and `"FR"`/`"France"`→France resolve.
- **`static func bestMatch(candidates: [Candidate], region: String?) -> Candidate?`** — empty list → `nil`; if `region == nil` → `candidates.first`; else first candidate satisfying `regionMatches`, or `nil` if none (INV-2 — NEVER fall through to top when a region was given). `isRegionToken(_:)` reuses the state table + a small country-code/country set + checking against admin1 isn't possible pre-geocode, so it keys off the known-token sets only (a place name like `"Francisco"` is correctly NOT a token). **NOTE (review NIT):** `isRegionToken` MUST recognize full state NAMES too — the VALUES of `stateAbbreviations` (e.g. `"texas"`, `"maryland"`), not only the abbreviation KEYS — so check BOTH sides of the table (plus the country-code/country set). That way `"Paris Texas"` parses to region `"Texas"` while `"San Francisco"` (`"Francisco"` in neither side) stays city `"San Francisco"`.
- T1's suite now compiles and (logically) passes.
- New file ⇒ `Run: xcodegen generate`, commit `project.pbxproj` (if not already done in T1's slice).
- `Run:` `xcodebuild -project MMail.xcodeproj -scheme MMail -configuration Debug build CODE_SIGNING_ALLOWED=NO`
- `Expected:` BUILD SUCCEEDED.
- **Files:** `MMail/Mail/WeatherQuery.swift` (new), `MMail.xcodeproj/project.pbxproj` (regen)

### T3 — Define `WeatherResult` and refactor `fetch` to the three outcomes via the seam (INV-3, INV-4)
Edit `MMail/Mail/WeatherService.swift`. Introduce the result enum and re-thread `fetch`:
- **`enum WeatherResult { case success(WeatherInfo); case notFound; case error }`** — one case per INV-4 outcome.
- **`static func fetch(city: String? = nil) async -> WeatherResult`** (was `-> WeatherInfo?`):
  - **Empty/nil city (IP path, unchanged behavior):** call `geolocate()` exactly as today (`WeatherService.swift:8-12,59-76`); on success build the forecast and return `.success`; on geolocate-nil or forecast-failure return `.error` (NOT `.notFound` — the model treats `.error` as "no change", which preserves prior weather and never alerts; INV-4/INV-5). `geolocate()` stays reachable ONLY here (INV-3).
  - **Non-empty city (typed path — NEVER calls `geolocate()`, INV-3):** `let parsed = WeatherQuery.parse(city)`; geocode by `parsed.city` ALONE (fixes the whole-string bug, spec line 28) with `count` ≥ 10 (replace `count=1` at `WeatherService.swift:45`). If the geocode request fails / can't decode (the `try?`/`guard` at `WeatherService.swift:49-52`) → `return .error`. Map the decoded results array → `[WeatherQuery.Candidate]`. `guard let chosen = WeatherQuery.bestMatch(candidates: candidates, region: parsed.region) else { return .notFound }` (region-unmatched or empty list → `.notFound`). Build the label from `chosen.name` + `admin1`/`country_code` (preserving `WeatherService.swift:53-55` logic). Run the FORECAST request (the existing `WeatherService.swift:14-37` block, factored to take lat/lon/label); if it fails/can't decode (`WeatherService.swift:24-26`) → `return .error` (the post-bestMatch forecast-failure sub-case, INV-4 — same `.error` as a geocode failure, NEVER `.notFound`, NEVER IP fallback). On success → `return .success(weatherInfo)`.
- Keep `condition(for:)` and the forecast-building math byte-identical (Non-Goals: no forecast/`WeatherInfo` change). `geocode` is folded into `fetch`'s typed branch (or kept as a helper returning `[Candidate]`); either way it MUST NOT call `geolocate()`.
